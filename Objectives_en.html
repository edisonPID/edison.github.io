Project Objectives

OG1.- Detect anomalous behavior in SDN networks early With this general objective, the early detection of anomalies in the network is sought through a machine learning algorithm. From the processing of the traffic captured in the infrastructure layer, the necessary information will be provided to the algorithm so that it can detect anomalous traffic and, if possible, classify it. This process must be efficient enough to be useful for the system, so apart from the precision in the detection, the amount of information necessary to obtain said precision will be evaluated, trying to keep this amount minimal.

OG2.- Act on the control plane of SDN networks to modify their behavior and mitigate possible threats Directly related to the previous objective, and as a subsequent step, the development of a solution is proposed that, by interacting with the SDN controller, manages to modify the treatment that the infrastructure gives to data flows, with the objective of mitigating future threats based on to the information obtained in the classification process carried out in OG1.

The specific objectives that will allow the aforementioned general objectives to be achieved are the following:

OE1.- Characterize the different particularities of existing SDN solutions and how they impact the detection of anomalous behavior. Due to the standardization and open nature of this type of solutions, in recent years new technologies related to the SDN ecosystem have appeared. In this way, in the components of an SDN scenario there are alternatives that can impact the general objective. SDN devices communicate with the controller through an API, and have structures to process the frames. The controller also provides two APIs, one to communicate with devices (southbound) and another to communicate with applications (northbound). Controller and applications allow you to implement decision making. In fact, the driver chosen largely determines the set of applications available. The controller can use various southbound APIs and protocols to manage the devices, and the API chosen will be determined mainly by the API supported by the devices themselves. In addition, there are also various alternatives for the northbound API. Regarding the application layer, although SDN applications are of great importance as they allow the control logic of SDN devices to be managed to a large extent, there is currently no standard. This causes controllers to often provide repositories where developers can share their software, which means that there may be multiple different implementations with the same purpose. A network emulation environment will be selected, after the corresponding analysis of available options. This environment will allow the different solutions to be validated and tested. With this specific objective, these decision making that may impact OG1 will be analyzed, and the characteristics of the existing solutions will be detailed.

OE2.- Evaluate different alternatives for early detection of anomalies in SDN environments As already mentioned in the background section, there are many models that have been applied to analyze the behavior of traffic in a network. Each of these models and algorithms has its peculiarities. Consequently, it is necessary to know the alternative in question to adapt the data entry to its characteristics. Hence, data engineering is of great relevance in this context and particularly feature engineering. Once the frames are captured, the characteristics must be defined, each of them treated according to the needs of the algorithms to be used and the appropriate ones selected. In this way, we seek to make the most of the capabilities of the algorithms. Specifically, machine learning algorithms will be used in this project. The algorithms will be experimentally validated against a dataset generated in a controlled environment and, if possible, against public datasets.

OE3.- Define and validate new metrics in the early detection of anomalies Traditional anomaly detection metrics are not appropriate for evaluating solutions where time is important. Although there are already some metrics that allow us to evaluate the importance of time in decision making, they present various drawbacks. This objective addresses the need for new metrics that penalize those proposals that, although accurate, are not carried out in time. These metrics must be limited and will consider different penalties for errors in detecting anomalies and errors in detecting normal traffic.

OE4.- Analyze the different possibilities of interaction with the controller Currently, although there are different implementations of SDN controllers, they all have the same general architecture: infrastructure, controller and application. The communication path defined to exchange data with the controller from an application is the northbound API, which provides developers with communication interfaces and services. Depending on the controller selected, these programming interfaces will be different and, in some cases, a single controller may even provide several specific interaction methods. Furthermore, it is necessary to take into account that, in the case of open source controllers, it is also possible to modify the behavior of said controller by developing complete functional units, building a new vertical service. An inventory of the different connection methods with the controller will be made, analyzed and the most appropriate one for this project will be chosen.

OE5.- Analyze the alternatives for modifying the behavior of the network In most SDN approaches, switches make decisions based on rules, which typically contain the following information: input port, MAC addresses, IP addresses, and ports. Based on these characteristics that define a flow, the actions that switches can perform are switching the packet to a specific interface, encapsulating and sending data to the controller, discarding the traffic, or switching through the conventional switch switching process. . The controller tells the switches what actions should be taken with each flow and, therefore, an application that knew the characteristics of the anomalous flows could communicate with the controller to change the actions associated with those flows.

OE6.- Design and implementation of an interconnection system between the anomalous behavior detection module and the SDN controller. As a result of the work developed in OG1, a classification algorithm optimized for the environment and a classified data set are obtained. Based on this classification of the dataset flows, it will be determined what actions (switch over a specific port, discard, encapsulate and send to the controller, etc.) should be carried out with the new flows received in the SDN, whose characteristics are similar to those of anomalous flows. To do this, it will be necessary to load a set of classified data, define the rules for assigning new rules, generate and transmit these changes to the controller.

OE7.- Evaluate the behavior of the SDN network under anomalous conditions. This objective aims to conclude whether early anomaly detection techniques are effective in mitigating threats in an SDN network. For this, different tests will be carried out and the results obtained will be evaluated.
